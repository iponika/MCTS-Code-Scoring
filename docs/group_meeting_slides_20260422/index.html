这里把系统路线图压缩成四步。第一步是统一评分标准，否则不同数据集的分数没法混在一起。第二步是 MCTS 数据生成，探索不同思考路径并标注节点价值。第三步是训练，policy 负责输出评分和证据，value head 负责评估候选路径。第四步是 value-guided 推理。

现在最小闭环已经跑通，问题不再是“能不能跑”，而是“哪一段真正有效，以及怎么评测才公平”。

scoring semantics

为什么用 AXIOM，而不是简单 0-100？

AXIOM 是 ordinal scale：核心语义是“修到可部署状态所需工作量”。先判断功能正确性，再细分修复/重构成本。

0

严重不匹配或根本错误，重写比修复更划算。

1

功能有缺陷，需要较大修复。

2

功能有缺陷，小修可恢复。

3

功能正确，但需要较大质量重构。

4

功能正确，仅需小幅代码质量调整。

5

Production-ready，无需修改。

0-2：功能不正确 / 不匹配
3-5：功能正确 / 质量分层

AXIOM 的好处是它给每个分数档明确语义，不是简单的一到五星。它的核心边界是 correctness：0 到 2 表示功能不正确或者不匹配；3 到 5 表示功能正确，然后再根据代码质量和修复工作量细分。

这也决定了我们的评测不能只看 MAE，还要看 boundary accuracy、low-FP 和 high-FN。因为把正确代码打成错误，和把错误代码打成正确，都是比差一分更严重的问题。

data alignment

训练数据：先对齐标签语义

数据源当前定位标签处理
CodeCriticBench主训练/评测数据之一correctness + score 映射到 AXIOM 0-5
AXIOMBench评分语义锚点原始 0-5 直接作为 AXIOM grade
Code-DiTing弱正确性监督只作为 0-2 / 3-5 区间标签
CodeJudgeBench相对排序监督候选更适合 pairwise ranking，短期低权重
原则：不要把所有数据集粗暴线性映射到同一个 0-100 回归目标。

训练数据里目前主要有这四类。CodeCritic 和 AXIOM 是当前最可靠的精确或近似精确标签。Code-DiTing 是二分类，只能说明大致正确或错误，不应该直接当成 0 或 5。CodeJudgeBench 是正负样本对，天然更适合排序训练。

这里最重要的技术选择是：先对齐标签语义，再训练。否则模型学到的只是不同数据集的平均打分风格，而不是统一评分原则。

effective choices

目前看有效的技术选择

去掉模型自评分

不再让基础模型给自己的思考节点打分，避免把模型偏见直接写入训练标签。

Verifier value-only

verifier 发现无依据证据时，只作为 value 负样本，不强迫 policy 模仿突变修正文本。

response_mean_value

候选选择不再只看最后 token 的 value，而看 response token 平均 value。

这些选择的共同点：让 value 更像“证据质量和评分可靠性”的监督，而不是模型自我感觉。

这里有三个目前看比较有效的选择。第一，去掉模型自评分。早期想过让模型给自己打分，但这会把基础模型的偏见写进标签，现在已经改成客观标签和 verifier 约束。

第二，verifier correction 改成 value-only。如果强迫 policy 学 verifier 修正后的文本，输出会变得不自然，所以只把无依据路径作为 value 负样本。

第三，value-guided 选择从 last_value 改为 response_mean_value。最后 token 很容易掩盖前面低置信的判断，平均 value 更稳定。

这是目前最重要的正结果。评测是 40 条 CodeCritic 保留集，value-guided MCTS 是三次 seed 的平均。

相对基础模型直接评分，MAE 从 1.450 降到 1.089，boundary accuracy 提升到 0.804，把错误代码评高的 high-FN 从 0.200 降到 0.071，无依据证据也明显减少。

但它不是全指标胜利。low-FP 从 0.025 升到 0.125，说明 value-guided 有保守偏置，更容易把正确代码打低。

ablation / candidate selection

value-guided 的选择指标很关键

score_keyMAEboundaryunsupported
last_value1.3160.7370.026
response_mean_value1.0810.7570.027
conservative_value1.4100.6670.000

解释

`last_value` 只看最后 token，可能掩盖同一步思考中前面的低置信判断。

`response_mean_value` 对整段 response 求平均，更适合选择稳定候选。

过度保守会减少 unsupported，但会伤害 MAE 和边界判断。

这里是一个关键消融。最开始我们用 last_value，也就是 response 最后一个 token 的 value。后来发现它容易掩盖前面判断中的问题。

改成 response_mean_value 后，MAE 和 boundary 都明显更好。conservative_value 虽然把 unsupported 降到 0，但代价是 MAE 和 boundary 变差。所以现在默认用 response_mean_value。

结果是负的。strict valid rate 下降，value-guided MCTS 最差；而且 low-FP 很高，说明模型容易把正确代码打到 0 到 2。

所以这里的结论是：当前模型还没有证明通用评分能力。它更像是适应了 CodeCritic 分布。

technical difficulties

目前需要汇报请教的难点

泛化能力

同分布有效，但 AXIOM 保留集失败。需要证明模型学到的是评分原则，不是数据集风格。

保守偏置

value-guided 降低 high-FN，但提高 low-FP。评分任务到底应如何权衡两类错误？

长样本显存

4090 上 Qwen3.5-9B + value head + 1152 token 已经很紧。双卡 DDP 不能合并单样本显存。

正在尝试：短训练 prompt 把保留训练样本从 130/1360 提高到 640/1360，但长样本仍是瓶颈。

这页是希望老师和同学给建议的地方。第一是泛化，尤其是怎么设计实验证明学到的是评分原则。第二是风险偏好，value-guided 现在偏保守，这在安全场景可能可以接受，但在评分场景会影响校准。第三是资源限制，长样本训练显存压力很大，普通双卡 DDP 不能解决单样本 OOM。

我们已经用短训练 prompt 把样本保留量从 130 条提高到 640 条，但这还不是最终解决方案。

evaluation design

后续评测：怎样才让人信服？

固定对照组

Base Direct、Trained Direct、Base MCTS / Self-MCTS、Trained Value-Guided MCTS、Supervised MCTS 上限。

固定测试套件

CodeCritic held-out、AXIOM held-out、Code-DiTing boundary、CodeJudgeBench pairwise。每次训练都同时报告。

指标组合

valid_rate、MAE、median_err、boundary_acc、low-FP、high-FN、unsupported、pairwise_acc、repeat std。

公平性约束

MCTS 与 direct 要控制 token / 采样预算；如果 MCTS 多采样，应增加 direct 多采样 rerank baseline。

后续评测我建议固定四类对照和四类测试集。每次训练都跑同一套指标，避免只展示有利数据集。

公平性上，MCTS 如果用了更多 token 或更多采样，就不能只和 direct 单次询问比较。最好增加 direct 多采样 rerank baseline，或者明确报告推理成本。
